# バージョニング方針

このリポジトリでは Semantic Versioning 2.0.0（SemVer）に従います。

以下は簡潔な実務向けの要点です。詳細は公式仕様を参照してください：

[Semantic Versioning 2.0.0 specification](https://semver.org/)

## 短いまとめ（実務向け）

- 形式: `MAJOR.MINOR.PATCH`（例: `1.4.2`）
- **MAJOR**: 後方互換性を壊す変更を行ったときに上げる。
- **MINOR**: 後方互換な機能追加を行ったときに上げる。
- **PATCH**: 後方互換なバグ修正を行ったときに上げる。

例:

- バグ修正 → `PATCH` を上げる: `1.0.0` → `1.0.1`
- 後方互換な機能追加 → `MINOR` を上げる: `1.0.1` → `1.1.0`
- 互換性を破る変更 → `MAJOR` を上げる: `1.1.0` → `2.0.0`

## プレリリース / ビルドメタデータ（任意）

- プレビュー用に `1.2.0-alpha.1` のようなプレリリース識別子を使えます。
- `+` 以下のビルドメタデータはバージョンの優先度には影響しません。

## このリポジトリでの運用方法

- 公開対象は `packages/motion-plus-installer/package.json` です。
- 公開手順（ワークフロー利用）:

  1. `packages/motion-plus-installer/package.json` の `version` を更新する。
  2. 変更をコミットし、`main` へプルリクエストを作成・マージする。
  3. マージ後、GitHub Actions の publish ワークフローが自動でビルドと公開を行います。

- 代替手順: ローカルで `git tag v1.2.3` を作成して `git push origin v1.2.3` すると、タグプッシュによりワークフローが起動します。

## すばやいチェックリスト（リリース作業）

1. `packages/motion-plus-installer/package.json` の `version` を適切に更新する。
2. 変更をコミットして PR を作成、レビュー後に `main` にマージする。
3. マージ後に自動でビルドと公開が実行されることを確認する。
4. 必要なら GitHub Release を作成してリリースノートを残す。
